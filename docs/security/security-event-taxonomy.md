# DailyPage Security Event Taxonomy

## Purpose

This document defines the initial security event taxonomy for DailyPage.org.

DailyPage should emit structured application-layer security events for behavior that is important to authentication, authorization, abuse prevention, moderation, and privileged actions.

The goal is to create a stable vocabulary for future security logging, detection, investigation, dashboards, and possible Wazuh ingestion.

This document is intentionally limited to event names and event families. The companion schema and implementation documents should define the JSON structure, sensitive-data handling rules, and staged implementation plan.

## Scope

This taxonomy applies to security-relevant events generated by the DailyPage application itself.

It does not replace:

- Nginx access logs
- System authentication logs
- Wazuh agent logs
- MongoDB logs
- Application error logs

Instead, this taxonomy covers events where DailyPage has application-specific context that lower-level logs do not have.

Examples:

- A login attempt failed because the password was incorrect.
- A login was blocked because the user's email was not verified.
- A password reset token was invalid or expired.
- A comment was blocked by a rate limit.
- A user attempted to edit content they did not own.
- A moderation report was submitted.
- An admin performed a privileged action.

## Design Principles

DailyPage security events should be:

### Structured

Events should be emitted as structured JSON objects, not free-form strings.

### Stable

Event names should be consistent over time so searches, dashboards, and detections do not break unnecessarily.

### Specific

Event names should be specific enough to support investigation.

Prefer:

- `auth.login.failure`
- `password_reset.token_expired`
- `authz.denied.not_owner`

Avoid vague names like:

- `security.error`
- `auth.problem`
- `user.failed`

### Safe

Security events must not log secrets, plaintext passwords, reset tokens, verification tokens, session cookies, JWTs, or full private user-generated content.

### Investigation-Oriented

Events should help answer:

- What happened?
- Who or what was involved?
- Was the action successful, failed, blocked, or denied?
- Why did it happen?
- What account, object, or route was targeted?
- Can this event be correlated with other events?

### Implementation-Neutral

This taxonomy should not require a specific SIEM, logging backend, or alerting platform. The same event names should be usable in local JSONL logs, Wazuh ingestion, future dashboards, or sanitized portfolio examples.

## Naming Convention

Event names should use lowercase dot-separated strings.

Recommended pattern:

```text
<family>.<action>[.<detail>]
```

Examples:

```text
auth.login.success
auth.login.failure
password_reset.token_expired
rate_limit.comment.blocked_ip
moderation.report.submitted
admin.action.content_hidden
authz.denied.not_owner
```

Guidelines:

- Use nouns for event families.
- Use clear verbs or action words for the middle segment.
- Use a third segment when it clarifies the reason or target.
- Avoid spaces, uppercase letters, and punctuation other than dots and underscores.
- Keep names readable by humans.

## Event Families

The initial taxonomy contains six event families:

1. Authentication events
2. Password reset events
3. Rate limiting and abuse-control events
4. Moderation and reporting events
5. Admin and privileged action events
6. Authorization and access-denied events

## Authentication Events

Authentication events describe login, logout, registration, and email verification behavior.

### Event Types

```text
auth.login.success
auth.login.failure
auth.login.blocked_unverified_email
auth.logout.success
auth.register.success
auth.register.rejected_username
auth.email_verification.sent
auth.email_verification.success
auth.email_verification.failure
```

### Event Meanings

| Event Type | Meaning |
|---|---|
| `auth.login.success` | A user successfully logged in. |
| `auth.login.failure` | A login attempt failed. |
| `auth.login.blocked_unverified_email` | A login attempt was blocked because the account's email was not verified. |
| `auth.logout.success` | A user successfully logged out. |
| `auth.register.success` | A new user account was successfully registered. |
| `auth.register.rejected_username` | A registration attempt was rejected because of a disallowed or invalid username. |
| `auth.email_verification.sent` | An email verification message was sent. |
| `auth.email_verification.success` | A user successfully verified their email address. |
| `auth.email_verification.failure` | An email verification attempt failed. |

### Notes

Authentication events should support investigation of brute-force attempts, account enumeration attempts, suspicious login patterns, unverified account behavior, and registration abuse.

Authentication events must not log plaintext passwords.

For failed login events, DailyPage may not know a valid `user_id`. In that case, the event may identify the attempted account using a safe field such as a normalized username, username hash, email hash, or null value, depending on the final schema rules.

## Password Reset Events

Password reset events describe account recovery activity.

### Event Types

```text
password_reset.requested
password_reset.email_sent
password_reset.token_validated
password_reset.token_invalid
password_reset.token_expired
password_reset.completed
password_reset.rate_limited
```

### Event Meanings

| Event Type | Meaning |
|---|---|
| `password_reset.requested` | A password reset was requested for an account or submitted email address. |
| `password_reset.email_sent` | A password reset email was sent. |
| `password_reset.token_validated` | A password reset token was presented and accepted as valid. |
| `password_reset.token_invalid` | A password reset token was invalid, malformed, reused, or not found. |
| `password_reset.token_expired` | A password reset token was recognized but expired. |
| `password_reset.completed` | A password was successfully reset. |
| `password_reset.rate_limited` | A password reset request was blocked by a rate limit. |

### Notes

Password reset events are high-value because password reset flows are closely related to account takeover risk.

These events must never log:

- Password reset tokens
- Password reset links
- New passwords
- Old passwords
- Session cookies
- JWTs

For privacy-preserving investigation, events may use a user ID, email hash, or submitted identifier hash depending on what information is available and safe to log.

## Rate Limiting and Abuse-Control Events

Rate limiting events describe actions blocked or constrained because of abuse-prevention rules.

### Event Types

```text
rate_limit.comment.blocked_user
rate_limit.comment.blocked_ip
rate_limit.comment.blocked_link_ip
rate_limit.auth.blocked_ip
rate_limit.password_reset.blocked_ip
```

### Event Meanings

| Event Type | Meaning |
|---|---|
| `rate_limit.comment.blocked_user` | A logged-in user's comment attempt was blocked by a user-based comment rate limit. |
| `rate_limit.comment.blocked_ip` | A comment attempt was blocked by an IP-based comment rate limit. |
| `rate_limit.comment.blocked_link_ip` | A comment containing a URL was blocked by a link-specific IP-based rate limit. |
| `rate_limit.auth.blocked_ip` | An authentication attempt was blocked by an IP-based authentication rate limit. |
| `rate_limit.password_reset.blocked_ip` | A password reset attempt was blocked by an IP-based password reset rate limit. |

### Notes

DailyPage already has comment rate limiting logic. That makes comment-related rate limit events a strong first candidate for future instrumentation.

Rate limit events are useful for detecting:

- Spam attempts
- Link spam
- Brute-force login behavior
- Password reset abuse
- Automated probing
- Repeated abuse from the same source

Rate limit events should include enough structured context to support correlation without exposing unnecessary raw personal data.

## Moderation and Reporting Events

Moderation events describe user reports and moderation actions.

### Event Types

```text
moderation.report.submitted
moderation.report.rejected_invalid
moderation.report.rate_limited
moderation.report.reviewed
moderation.content.hidden
moderation.content.restored
moderation.content.deleted
```

### Event Meanings

| Event Type | Meaning |
|---|---|
| `moderation.report.submitted` | A user submitted a report about content or behavior. |
| `moderation.report.rejected_invalid` | A report submission was rejected because it was invalid. |
| `moderation.report.rate_limited` | A report submission was blocked by a rate limit. |
| `moderation.report.reviewed` | A report was reviewed by an authorized user or admin process. |
| `moderation.content.hidden` | Content was hidden as a moderation action. |
| `moderation.content.restored` | Previously hidden content was restored. |
| `moderation.content.deleted` | Content was deleted as a moderation action. |

### Notes

Security logs should record that a moderation event occurred, but they should not duplicate the full text of user-generated reports, comments, or blocks.

The application database should remain the source of truth for full moderation report contents. Security events should contain references and reason codes, not full private text.

## Admin and Privileged Action Events

Admin events describe privileged access and privileged changes.

### Event Types

```text
admin.login.success
admin.login.failure
admin.action.user_role_changed
admin.action.content_hidden
admin.action.content_deleted
admin.action.report_resolved
admin.action.settings_changed
admin.access.denied
```

### Event Meanings

| Event Type | Meaning |
|---|---|
| `admin.login.success` | An admin user successfully logged in or accessed an admin session. |
| `admin.login.failure` | An admin login attempt failed. |
| `admin.action.user_role_changed` | A user's role or privileges were changed. |
| `admin.action.content_hidden` | An admin hid content. |
| `admin.action.content_deleted` | An admin deleted content. |
| `admin.action.report_resolved` | An admin resolved a moderation report. |
| `admin.action.settings_changed` | An admin changed security-relevant or site-level settings. |
| `admin.access.denied` | A request for an admin-only resource was denied. |

### Notes

Admin events should be treated as high-signal because privileged actions should be relatively rare.

Future detections may watch for:

- Admin access failures
- Admin actions from unusual sources
- Role changes
- Content deletion
- Settings changes
- Admin access attempts by non-admin users

## Authorization and Access-Denied Events

Authorization events describe denied attempts to access or modify resources.

### Event Types

```text
authz.denied.not_logged_in
authz.denied.unverified_email
authz.denied.not_owner
authz.denied.not_admin
authz.denied.invalid_csrf
authz.denied.invalid_token
```

### Event Meanings

| Event Type | Meaning |
|---|---|
| `authz.denied.not_logged_in` | A request required login, but the requester was not authenticated. |
| `authz.denied.unverified_email` | A request required verified email, but the user's email was not verified. |
| `authz.denied.not_owner` | A user attempted an action on content or an object they did not own. |
| `authz.denied.not_admin` | A user attempted an admin-only action without admin privileges. |
| `authz.denied.invalid_csrf` | A request was denied because of an invalid or missing CSRF token. |
| `authz.denied.invalid_token` | A request was denied because an application token was invalid. |

### Notes

Authorization-denied events are useful for detecting suspicious object access attempts, permission-boundary probing, CSRF issues, and unauthorized admin-route access.

Repeated `authz.denied.not_owner` events may indicate a user or automated client is probing object IDs or attempting unauthorized modification.

## Suggested Initial Instrumentation Candidates

The first implementation phase should not attempt to instrument every event type at once.

Good initial candidates:

```text
auth.login.success
auth.login.failure
auth.login.blocked_unverified_email
password_reset.requested
password_reset.completed
password_reset.token_invalid
password_reset.token_expired
rate_limit.comment.blocked_user
rate_limit.comment.blocked_ip
rate_limit.comment.blocked_link_ip
moderation.report.submitted
authz.denied.not_owner
authz.denied.not_admin
```

These events are useful, concrete, and closely tied to existing DailyPage functionality.

## Change Control

Changes to this taxonomy should be made carefully because event names may become dependencies for saved searches, dashboards, alerts, tests, and documentation.

When adding or changing event names:

1. Prefer adding new event types rather than changing existing ones.
2. Avoid renaming existing events unless necessary.
3. Document deprecated event names if any are replaced.
4. Update schema examples and implementation tests when needed.
5. Avoid event-name churn.
